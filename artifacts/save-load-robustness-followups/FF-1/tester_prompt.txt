HOW TO MUTATE A VALID SAVE'S FLEET REFERENCE ID TO AN OUT-OF-RANGE VALUE
(two viable approaches; pick whichever fits each case best):

Approach A -- byte-patch the serialized location field (recommended for case 1,
matches the existing tests' byte-surgery style):
  FFleet::save() (src/strategic/FFleet.cpp) writes its fields in this fixed order:
    writeU32(m_ID)            -> 4 bytes
    writeString(m_name)       -> 4-byte little-endian length prefix + name bytes
    writeU32(m_owner)         -> 4 bytes
    writeU32(m_location)      -> 4 bytes   <-- PATCH THIS for case 1
    write(m_inTransit)        -> 1 byte (bool)
    writeU32(m_destination)   -> 4 bytes
    write(m_transitTime)      -> 4 bytes (unsigned int)
    write(m_jumpLength)       -> 4 bytes (unsigned int)
    write(m_speed)            -> 4 bytes (unsigned int)
    writeU32(m_jumpRouteID)   -> 4 bytes   <-- PATCH THIS for case 2
    ... (icon string, militia bool, home string, holding bool, floats, ship count)
  Locate a specific placed fleet by scanning buildValidSaveBytes() for its name's
  length-prefixed tag -- exactly as the existing test at ~lines 314-323 finds a
  fleet name tag (it builds the 4-byte LE length prefix followed by the ASCII
  name bytes; e.g. "Task Force Prenglar" has length 19). From the byte just AFTER
  the located name tag, skip 4 bytes (m_owner) to reach the 4-byte m_location
  field; overwrite those 4 bytes with a LE uint32 that is neither 0 nor any real
  loaded system ID (systems are a small set; 0xFFFFFF00 or 0x0000BEEF is safely
  out of range). To patch m_jumpRouteID instead, skip a further 1 + 4 + 4 + 4 + 4
  = 17 bytes past the location field, then overwrite the next 4 bytes with a LE
  uint32 that is neither FFleet::NO_ROUTE (0xFFFFFFFF) nor any real route ID
  (e.g. 0x0000BEEF). Prefer the name of a fleet you KNOW is placed in a system
  (e.g. "Task Force Prenglar" or "Task Force Cassidine") so the location field is
  a real nonzero system ID before you clobber it.
