Implementation context (for documentation accuracy):
- New helper `int FBattleDisplay::measureWrappedActionPromptWidth(wxDC&, const wxString&, int maxWidth) const`
  (declared in include/tactical/FBattleDisplay.h, defined in src/tactical/FBattleDisplay.cpp
  immediately after drawWrappedActionPrompt()) mirrors the greedy word-wrap algorithm used by
  countWrappedActionPromptLines()/drawWrappedActionPrompt(), returning the widest wrapped line's
  pixel width instead of a line count.
- drawMoveShip() now computes `instrW` as the max of measureWrappedActionPromptWidth() over
  turnPrompt, detailPromptOne, detailPromptTwo, and detailPromptThree (all measured against
  promptMaxWidth), instead of dc.GetTextExtent() on the raw unwrapped strings. This was the root
  cause fix.
- buildMovePromptText() (src/tactical/FBattleDisplay.cpp, declared in include/tactical/FBattleDisplay.h)
  gained a 4th output parameter `wxString & detailPromptThree`. The former single long line
  "Press the 'Movement Done' button when all ships have been assigned their movement instructions."
  is now split into detailPromptTwo = "Press the 'Movement Done' button when all ships" and
  detailPromptThree = "have been assigned their movement instructions." in the normal-movement
  branch; detailPromptThree is cleared to empty in the other two branches (stopped-facing-selection
  and no-ship-selected).
- drawMoveShip() now calls buildMovePromptText() directly instead of duplicating the
  prompt-construction logic inline (deduplication), so refreshMovePromptReservation() and
  drawMoveShip() can never diverge on prompt text or line count.
- Caption: drawMoveShip() draws a 4-line caption directly above the Turn Left/Turn Right button
  panel using a smaller 8pt font (wxFONTFAMILY_SWISS, normal):
  "If at the end of a ship's movement" / "you want to make a facing change" /
  "(and have remaining MR), use these" / "buttons to make a single final turn." -- exact wording
  as specified by the task, hardcoded into 4 lines rather than dynamically wrapped so the line
  count and per-line width are deterministic. Caption width is measured via dc.GetTextExtent()
  per line; blockWidth = max(button-panel best width, captionW) is used for the fit check against
  the ship-stats column; the caption's top-left is at (panelX, getActionPromptLineY(0)); the button
  panel itself is placed captionH + BORDER below that.
- Panel growth: before the fit check, drawMoveShip() computes blockBottom = panelY +
  button-panel-height + BORDER and, if that exceeds m_lowerPanelLayoutState.requestedDisplayHeight,
  grows requestedDisplayHeight and calls applyRequestedDisplayHeight() so the lower panel grows to
  fit caption + buttons without clipping.
- Fit/visibility guard unchanged in spirit: the panel (and caption) are only drawn/shown when
  panelX + blockWidth + 2*BORDER <= statsLeft (statsLeft = shipStatsLeftMargin when initialized).
  Because instrW is now bounded by actual wrapped line widths, lMargin no longer overshoots
  statsLeft=300 at the default window size, so the guard now passes.
- Untouched: Movement Done button logic, onMoveDone()/onTurnLeft()/onTurnRight() handlers,
  canApplyEndOfMoveTurnLeft()/canApplyEndOfMoveTurnRight() enable-state wiring,
  drawCurrentShipStats(), and all other draw*() phase methods and their instruction text/columns.
