WARNING
- src/gui/WXPlayerDisplay.cpp:26-88 - WXGameDisplay::draw() (src/gui/WXGameDisplay.cpp:33-38) reaches FMap through a second, unguarded path -- WXPlayerDisplay::drawFleets() -- that FR-2 did not guard and could not have guarded within its declared scope.
  Independently verified: WXGameDisplay::draw() calls playerDisplay.drawFleets(dc, *itr) for every non-NULL player (src/gui/WXGameDisplay.cpp:37). drawFleets() unconditionally executes `FMap *map = &(FMap::getMap());` (WXPlayerDisplay.cpp:28, no FMap::hasMap() guard) and, when the player has at least one fleet, dereferences it via map->getJumpRoute() (line 46) and map->getSystem() (lines 61-62, 81-82). If FMap::m_map were NULL while a player held a located fleet, this would reproduce the exact null-pointer-dereference crash class SF-nullfmap-paint-guard targeted. I independently traced the reachability argument the Security review relied on and confirmed it holds in the current codebase: FGame::init() calls initMap() (src/strategic/FGame.cpp:98) before initFleets() (line 111), and FGame::load() calls FMap::create() (line 698) and aborts before any player is constructed if map load fails (load() returns at line 701, well before the player loop starting at line 708) -- so under all current code paths a player with a located fleet cannot exist unless FMap already exists. This is a real but implicit ordering invariant, not an explicit guard, so it is fragile to future refactors of FGame::init()/load() or new fleet-construction paths, and there is no test that would catch a regression there. src/gui/WXPlayerDisplay.cpp is confirmed absent from FR-2's declared allowed-files list, so the Implementer could not have closed this gap without an unauthorized scope expansion; that is a legitimate, upstream scoping decision, not an Implementer defect. I classify this WARNING rather than BLOCKING for FR-2 specifically: the acceptance criteria as handed to this Verifier are anchored to the concrete, documented crash (commit 09cc698a / test 92ae3f38), which is fully and behaviorally closed, and the residual drawFleets() path is currently unreachable given verified codebase invariants. Recommend opening a new deferred-task entry (e.g. a scoped drawFleets null-map guard, analogous to how SF-nullfmap-paint-guard itself was spun out of the P5-5 review) rather than reopening or blocking FR-2 on it.
