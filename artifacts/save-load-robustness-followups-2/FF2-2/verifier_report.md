- tests/strategic/FGameSaveFormatTest.cpp:911-922 - Tester judgment call (documented in-source and in the Tester's report) to omit a full-game 'truncated strictly inside a planet's station record' FGameSaveFormatTest case, on the grounds that it cannot behaviorally discriminate the FPlanet::load() fix from the unfixed code, independently verified CORRECT.
  Traced the call chain by reading the actual code (not just the claim): the shared read() helper (include/core/FPObject.h:196-200) sets istream failbit and returns nonzero on a short read regardless of whether the caller checks the return -- so even on unfixed FPlanet::load() (which ignores nested-read failures), a truncated station record leaves the underlying std::istream in a failed state. FSystem::load()'s planet loop (src/strategic/FSystem.cpp:103-110) only checks each FPlanet's own p->load(is) return, which unfixed code always reports as 0 -- but FMap::load() (src/strategic/FMap.cpp:340-357) issues a checked readU32(is,jumpCount) immediately after the systems loop completes, and that checked read observes the already-failed stream and returns nonzero, aborting FMap::load() and propagating to FGame::load()'s aggregate-abort path regardless of where inside a station record the truncation occurred. This holds for every truncation position (not just a favorable one), because C++ istream failbit persists across all subsequent operations until explicitly cleared, and there is always at least one later checked read (jumpCount, a subsequent system's own leading readU32, or FGame::load()'s own subsequent checks) downstream of any planet/station read. I independently reproduced this rather than relying on the Tester's or Security reviewer's written claim alone. The complementary unit-level test (tests/strategic/FPlanetTest.cpp:215-237, testLoadReturnsNonzeroOnTruncatedStationRecord) does genuinely discriminate the fix -- it calls FPlanet::load() in isolation with no downstream FMap/FSystem reads to mask the difference -- and I confirmed the fix-discriminating claim empirically (see review_scope_summary: unfixed code segfaults under the sibling unknown-station-type unit test, and the acceptance-relevant truncation unit test exercises the same nested-load-return-checking code path added by this fix). This is the correct behavioral-test-mandate-compliant substitute, not a coverage gap.
