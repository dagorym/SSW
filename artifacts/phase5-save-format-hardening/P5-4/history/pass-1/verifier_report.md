Findings

BLOCKING
- tests/strategic/FSystemTest.cpp:94-109 - FSystemTest::testLoadAdvancesNextIdPastLoadedID (AC4 for FSystem) is a vacuous/tautological test: it passes identically whether or not FSystem::load's H3 non-colliding guard (src/strategic/FSystem.cpp:116-118) is present.
  Empirically verified: in an isolated scratch git worktree (removed after the check, not part of this branch), the H3 guard block in FSystem::load was commented out, the strategic module and full tests/ suite were rebuilt, and `./SSWTests` from tests/ still reported OK (269 tests) -- this exact test did not fail. Root cause: the test default-constructs `FSystem s2;` (line 100) before calling `s2.load(buf)`; that default construction itself allocates the next sequential ID from the same shared static m_nextID counter (m_ID = ++m_nextID, same as the earlier `FSystem s(...)` construction), so m_nextID is already > loadedID before load() ever runs, making the guard's conditional branch always take the false path in this test and the final assertion `s3.getID() > loadedID` hold unconditionally regardless of the guard's correctness or presence. This violates AGENTS.md's mandatory Behavioral Verification policy ("A new or changed behavior is not considered covered until a behavioral test for it exists and fails against the unfixed code"): this test would not have failed against unfixed code, so AC4 for FSystem is not actually proven by test evidence, even though the implementation itself is correct on inspection. Sibling test FJumpRouteTest::testLoadAdvancesNextIdPastLoadedID (tests/strategic/FJumpRouteTest.cpp:158-179) demonstrates the correct pattern already used elsewhere in this same subtask: hand-craft a raw wire buffer with a deliberately huge ID (5000000) and call load() directly on a freshly-constructed object with no intervening default construction, so the assertion can only pass if the guard actually fires. The FSystem test should be rewritten the same way (hand-craft a stream with a large ID via readU32-compatible little-endian byte packing, matching FSystem::save's field order: ID, name, owner, coords, planet count) instead of relying on the real save()/load() round trip through a second default-constructed object. This is a test-coverage gap only; FSystem::load's H3 guard implementation itself (src/strategic/FSystem.cpp:116-118) was read directly and is logically correct (advances m_nextID to max(m_nextID, m_ID), consistent with the pre-increment m_ID = ++m_nextID allocation convention documented in the header).
