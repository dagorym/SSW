WARNING
- src/strategic/FPlayer.cpp:154,160,177 - FPlayer::load()'s m_unattached/m_fleets/m_destroyed loops discard the return value of v->load()/f->load(); FFleet::load() (src/strategic/FFleet.cpp:197) likewise discards v->load()'s return for ships. A stream truncated deep inside fleet/ship data therefore does not propagate failure up to FGame::load()'s per-player check (src/strategic/FGame.cpp:710), letting a corrupt save be silently accepted.
  Pre-existing gap in P5-3 scope (FPlayer.cpp/FFleet.cpp), not in P5-5's allowed file list (include/strategic/FGame.h, src/strategic/FGame.cpp, src/FMainFrame.cpp, include/FMainFrame.h) and not touched or worsened by the P5-5 diff. Confirmed accurate against the Security report's line references by direct source inspection. Bounded reads (readString capped at kMaxSerializedStringBytes, sticky failbit) mean the exploit surface is data-integrity/robustness, not memory corruption -- consistent with Security's LOW/MEDIUM disposition and CONDITIONAL PASS. Does NOT block P5-5's own acceptance; P5-5's own FGame::load() aggregation correctly checks every sub-load return it owns. RECORD AS FOLLOW-UP DEFECT (P5-3 scope): check FFleet::load()/FVehicle::load()/f->load() return values in FPlayer::load()'s three loops and the ship-load return in FFleet::load().
- src/FMainFrame.cpp:157,169 - onOpen() calls m_drawingPanel->setGame(m_game) (line 157, unchanged pre-P5-5 code) before load(); on a failed load, reportLoadError()->IStrategicUI::showMessage() opens a modal dialog whose nested event loop can deliver a paint to the now-mapped FGamePanel while FMap is still NULL, crashing in WXMapDisplay::getScale()/FMap::getMaxSize() (gui module, outside P5-5's allowed files). Both the Security stage and the Tester independently reproduced/documented this crash (StrategicGuiLiveTest.cpp:2323-2360 comment block, gdb backtrace).
  The underlying missing-NULL-guard defect lives entirely in gui-module draw code (FGamePanel::onPaint / WXGameDisplay::draw / WXMapDisplay::getScale), which P5-5 does not touch, so the fix is out of scope. Note for precision: the setGame()-before-load() call site that exposes this crash is pre-existing (unchanged context line in src/FMainFrame.cpp, not introduced by the P5-5 diff), but P5-5 is what newly makes the crash reachable in this flow -- before P5-5 there was no showMessage()/modal dialog at all on a failed load (the old code silently continued with a partially-built game instead). Does NOT block P5-5's stated acceptance criteria (onOpen's menu-disable / no-game-state behavior is correctly implemented and behaviorally verified without hitting this crash, by deliberately not showing the frame in the GUI test). RECORD AS FOLLOW-UP DEFECT: add a NULL-FMap guard in the gui draw path, or apply Security's suggested partial in-scope mitigation of deferring m_drawingPanel->setGame(...) in onOpen() until after a successful load.
